# 灵听 (Lingtin) - 产品需求文档

> 版本：v1.0 | 更新日期：2026-02-16

---

## 一、产品愿景与定位

### 定位

餐饮行业**语音智能管理平台**——以消费者反馈为核心驱动力，帮助门店建立"说了 → 记了 → 做了 → 验了"的管理闭环。

### 核心价值

1. **说了就记了**：语音自动转写 + AI 结构化，告别手写记录
2. **记了就做了**：AI 基于数据生成行动建议，推动问题落地解决
3. **做了就验了**：下一次桌访自动验证上次问题是否改善，形成闭环

### 解决的 3 个核心问题

| 问题 | 现状 | 灵听方案 |
|------|------|----------|
| 桌访数据流失 | 店长走完就忘，反馈靠记忆 | 语音自动记录 + AI 结构化存档 |
| 问题无人跟进 | 发现了问题但缺乏跟踪机制 | AI 自动生成行动建议 + 标记追踪 |
| 管理无法量化 | 不知道桌访做了多少、效果如何 | 覆盖率、情绪趋势、菜品热度等可视化看板 |

### 目标行业

餐饮为主（正餐、火锅、快餐），验证后扩展至酒店、零售等服务行业。

## 二、用户角色

| 角色 | 阶段 | 核心需求 | 使用场景 |
|------|------|----------|----------|
| **店长** | 当前 | 高效完成桌访、了解每日数据、跟进改善项 | 桌访录音、查看看板、处理行动建议 |
| 老板/CEO | 未来 v3+ | 多店宏观数据、AI 问答获取洞察 | AI 智库问答、跨店对比 |
| 区域经理 | 未来 v3+ | 多店巡查、汇总对比 | 巡店报告、区域数据看板 |
| 主管/领班 | 未来 v2+ | 参与例会、执行任务 | 例会纪要查看、任务接收 |
| 厨师长 | 未来 v2+ | 菜品反馈、改善跟踪 | 菜品维度数据查看 |
| 普通员工 | 未来 v4+ | 接收任务、反馈执行 | 任务列表、完成确认 |

> 当前阶段（v1.x）聚焦**店长**角色，在**单店内**跑通完整闭环。

## 三、核心闭环

### 消费者反馈驱动的管理闭环

```
① 桌访录音          ② AI 结构化分析        ③ 每日复盘/例会
   店长走桌 →          语音转写 →              查看数据 →
   录下对话             情绪/菜品/关键词          发现问题
      │                    │                      │
      │                    │                      │
      ▼                    ▼                      ▼
⑥ 再桌访验证        ⑤ 执行改善             ④ 生成行动建议
   下次桌访 ←          厨房调整 ←              AI 建议 ←
   自动对比             服务改进                待办清单
```

### 各环节定义

| 环节 | 输入 | 处理 | 输出 |
|------|------|------|------|
| ① 桌访录音 | 店长与顾客对话音频 | 本地录制 + 云端上传 | 音频文件 + 元数据（桌号、时间） |
| ② AI 分析 | 音频文件 | 讯飞 STT + AI 打标 | 转写文本、情绪分、关键词、菜品评价 |
| ③ 每日复盘 | 当日全部桌访数据 | 看板聚合展示 | 覆盖率、TOP 菜品、情绪概览 |
| ④ 行动建议 | 桌访分析结果 | AI 识别问题 + 生成建议 | 待办清单（建议行动项） |
| ⑤ 执行 | 待办清单 | 店长确认 + 分派执行 | 已处理/已忽略标记 |
| ⑥ 再验证 | 历史问题 + 新桌访数据 | 自动匹配对比 | 改善/未改善标记 |

## 四、应用场景

### 4.1 [P0] 桌访录音（已有 MVP，持续完善）

**场景描述**：店长在用餐时段走访各桌，与顾客简短交流，录音记录反馈。

**现有能力**：
- 选桌号 → 录音 → 自动上传 → AI 分析（STT + 打标）
- 数据看板（覆盖率、菜品 TOP5、情绪概览）
- AI 智库问答

**增强方向**：

#### 4.1.1 智能问卷提示

- 可配置的标准问题清单，录音时浮窗提示店长该问什么
- 问题示例：「请问您是怎么知道我们店的？」「今天是第几次来？」「对菜品口味满意吗？」
- 支持定期更换问题清单（按周/月配置）
- 独立功能项，不影响现有录音流程

#### 4.1.2 数据展示增强

- 桌访详情页：展示单条桌访的完整分析结果
- 历史趋势：情绪分、菜品评价的时间变化
- 对比视图：午市 vs 晚市、工作日 vs 周末

### 4.2 [P0] AI 建议行动清单

**场景描述**：基于当日桌访数据，AI 自动生成"建议行动项"，店长确认后执行。

**核心流程**：
```
当日桌访数据汇总
    ↓
AI 分析识别问题
    ↓
生成建议行动项
    • "3桌顾客反映酸菜鱼偏咸，建议与厨师长沟通调味"
    • "A5桌顾客对服务速度不满，建议优化出菜流程"
    ↓
店长查看清单
    ↓
标记：已处理 / 忽略（附原因）
```

**MVP 定义**：
- 不做完整任务系统（不分配、不设截止日、不验收）
- 仅生成建议 + 简单标记，先验证闭环价值
- 每日自动生成，也可手动触发

**数据结构要点**：
- 关联来源桌访记录
- 建议类型：菜品改善 / 服务改善 / 环境改善 / 其他
- 状态：pending / acknowledged / resolved / dismissed

### 4.3 [P1] 门店例会/复盘录音

**场景描述**：店长主持的班前会、班后复盘会，录音自动生成会议纪要。

**功能定义**：
- 会议类型选择：班前会 / 班后复盘 / 周例会
- 录音 → STT → AI 生成会议纪要
- 自动提取行动项（谁、做什么、截止时间）
- 关联当日/本周桌访数据作为讨论依据

**与桌访的区别**：
- 时长更长（5-15 分钟 vs 桌访 1 分钟）
- 多人对话 vs 一对一
- 输出为纪要文档 vs 结构化标签
- 需要说话人分离（后续增强）

### 4.4 [P2] 管理层会议

**场景描述**：区域经理/老板主持的跨门店管理会议。

**功能定义**：
- 多店数据自动汇报要点生成
- 跨店对比分析
- 会议纪要 + 跨店行动项

### 4.5 [P2] 巡店检查

**场景描述**：区域经理巡店，语音记录检查项并自动生成巡检报告。

**功能定义**：
- 巡检模板（卫生、陈列、服务标准等）
- 语音记录 → 自动填充检查项
- 生成巡检报告 + 整改任务
- 整改跟踪与复查

### 4.6 [P3] 全场景任务闭环系统

**场景描述**：从各场景（桌访、例会、巡检）自动产生的任务，统一管理。

**功能定义**：
- 完整任务生命周期：创建 → 分配 → 执行 → 验收
- 跨场景任务关联（桌访发现 → 例会讨论 → 巡检验收）
- 角色权限：谁可以创建/分配/验收
- 自动验收：通过桌访数据自动判断改善效果

## 五、通用技术底座

### 5.1 语音处理 Pipeline

所有语音场景共享同一处理管线，通过参数区分场景类型：

```
音频输入 → 格式标准化 → 讯飞 STT → AI 分析 → 结构化存储
              │                          │
              │                          ├── scene_type: table_visit
              │                          ├── scene_type: meeting
              │                          ├── scene_type: inspection
              │                          └── (按类型选择不同 prompt)
              │
              └── 短音频直传 / 长音频分片
```

### 5.2 AI 分析参数化

| 场景 | Prompt 关注点 | 输出结构 |
|------|---------------|----------|
| 桌访 | 菜品评价、服务感受、情绪 | 情绪分 + 关键词 + 菜品提及 |
| 例会 | 讨论要点、决议、行动项 | 纪要 + 行动项列表 |
| 巡检 | 检查项达标情况、问题 | 检查项评分 + 整改项 |

### 5.3 共享能力

- 统一的音频上传与存储服务
- 统一的 STT 调用封装
- 统一的 AI 分析调用封装（支持多模型切换）
- 统一的权限与认证体系

## 六、数据模型设计方向

### 现有表（保留并扩展）

| 表 | 当前状态 | 扩展方向 |

### 新增表方向

#### lingtin_action_items（行动建议表）- v1.x

```
id, restaurant_id, source_type, source_id,
category, suggestion_text, priority,
status (pending/acknowledged/resolved/dismissed),
resolved_by, resolved_at, dismiss_reason,
created_at
```

```
id, restaurant_id, template_name,
questions (JSONB), is_active,
effective_from, effective_to,
created_at
```

```
id, restaurant_id, meeting_type,

action_items (JSONB), participants (JSONB),
meeting_date, duration_seconds,
created_at
```

#### lingtin_inspection_records（巡检记录表）- v2.x

```

inspection_date,
created_at
```

#### lingtin_tasks（任务表）- v3.x

```
id, restaurant_id, title, description,
source_type, source_id,
assigned_to, assigned_by,
priority, status, due_date,
completed_at, verified_at, verified_by,
created_at
```

> 以上为设计方向，实际实现时再确认字段细节。

## 七、版本路线图

### v1.x — 桌访闭环（当前阶段）

**目标**：店长单店内跑通"录音 → 分析 → 行动 → 验证"闭环。

| 版本 | 内容 | 优先级 |
|------|------|--------|
| v1.1 | AI 建议行动清单 MVP | P0 |
| v1.2 | 智能问卷提示 | P0 |
| v1.3 | 桌访详情页 + 历史趋势 | P0 |
| v1.4 | 数据对比视图（午/晚、工作日/周末） | P1 |

### v2.x — 门店例会

**目标**：例会录音自动生成纪要 + 行动项，与桌访数据联动。

| 版本 | 内容 | 优先级 |
|------|------|--------|
| v2.0 | 例会录音 + 纪要生成 | P1 |
| v2.1 | 例会行动项提取 + 与桌访关联 | P1 |
| v2.2 | 说话人分离（增强） | P2 |

### v3.x — 多角色 + 巡店

**目标**：引入区域经理、老板角色；巡店场景上线。

| 版本 | 内容 | 优先级 |
|------|------|--------|
| v3.0 | 管理层会议支持 + 跨店对比 | P2 |
| v3.1 | 巡店检查模板 + 语音填报 | P2 |
| v3.2 | 角色权限体系完善 | P2 |

### v4.x — 全场景任务闭环

**目标**：所有场景产生的任务统一管理，自动化分派与验收。

| 版本 | 内容 | 优先级 |
|------|------|--------|
| v4.0 | 统一任务系统 | P3 |
| v4.1 | 自动验收（桌访数据回检） | P3 |
| v4.2 | 跨场景关联分析 | P3 |
