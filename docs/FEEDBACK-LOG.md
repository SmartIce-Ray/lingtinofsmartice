## 记录规范

### 编号体系

| 前缀 | 含义 | 示例 |
|------|------|------|
| `FB-` | 用户反馈（体验问题、使用障碍、正向评价） | FB-001 |
| `REQ-` | 功能需求（新功能请求、改进建议） | REQ-001 |

### 优先级定义

| 等级 | 含义 | 响应要求 |
|------|------|----------|
| **P0** | 阻断核心流程，用户无法完成关键操作 | 立即处理 |
| **P1** | 严重影响体验，但有替代路径 | 当前迭代处理 |
| **P2** | 改善体验，不影响核心流程 | 排入后续迭代 |
| **P3** | 锦上添花，长期优化方向 | 需求池积累 |

### 状态流转

```
待评估 → 已采纳 → 已排期 → 已上线
              ↘ 暂缓（附原因）
```

### 反馈来源标注

标明反馈者角色，便于理解用户视角：

- **店长** — 日常使用录音、看板、行动建议的一线用户
- **管理层** — 老板/区域经理，关注跨店数据和经营决策
- **店员** — 服务员/厨师，未来用户群体
- **顾客** — 终端消费者（间接反馈）
- **内部** — 团队成员在开发/测试中发现的产品问题

### 用户习惯与操作流程

**产品驱动 vs 报表模式**（2026-02-25 产品讨论沉淀）

- **报表模式（反模式）**：被动展示数据，靠用户自驱判断什么重要。典型特征：数字平铺、好差评混排、无行动出口
- **产品驱动模式（目标）**：系统主动呈现问题，引导用户决策。核心链路：**问题优先呈现 → 自带关键证据（顾客原话）→ 可听原声 → 有行动出口**
- 所有面向用户的新页面/改造必须遵循产品驱动原则，不做被动数据报表
- 好评和差评必须分区展示，不混排——用户不应自己从混合列表里挑问题
- 非菜品维度（服务速度、环境、态度）同等重要，不能只展示菜品
